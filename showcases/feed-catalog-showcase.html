Feed Catalog — Showcase

The "new feed" flow introduces a catalog step: choose a provider, then a feed type, then proceed to the existing feed creation form. Four layout variants for the catalog step.

Variant
Theme
Step

Variant A: Provider Grid

Recommended

Provider cards in a responsive grid. Each card shows the provider icon, name, and feed type count. Clicking drills into a list of feed types for that provider. Two-step flow: providers -> feed types -> form.

Why this variant: Progressive disclosure — the grid is scannable at a glance and providers with multiple feeds (GitHub: 3) justify the grouping. The breadcrumb provides clear back-navigation. The grid adapts naturally if more providers are added later.

Trade-off: Two clicks to reach the form. For providers with only one feed type (Vercel, HTTP, Shell), the intermediate step is unnecessary — could auto-skip to the form.

Variant B: Flat Grouped List

Simple

All feed types in a single scrollable list, grouped by provider with section headers. One click to reach the form. No intermediate drill-down. Best when the total number of feed types is small (under ~15).

Why this variant: Minimum clicks — everything is visible at once. The grouped sections give provider context without requiring navigation. Works well for the current ~7 feed types.

Trade-off: Gets long as more feed types are added. With 15+ types, scrolling becomes tedious and the flat list loses its advantage. No visual differentiation between providers beyond the section header.

Variant C: Split Pane

Dense

A nested split view within the settings main area. Provider list on the left, feed types for the selected provider on the right. Everything visible simultaneously — no navigation state to manage. Mirrors the sidebar pattern already used for settings sections.

Why this variant: Zero-click exploration — select a provider to see its types immediately. Familiar pattern (it mirrors the settings sidebar itself). Shows the feed type identifier (e.g., github-pr) which is useful for power users who write TOML configs directly.

Trade-off: Nested split-pane inside an already split-pane layout creates visual complexity. With only 5 providers, the left column feels sparse. The settings window (780px) may feel cramped with three columns of content.

Variant D: Accordion

Compact

Provider cards that expand inline to reveal feed types. No page navigation — everything stays in one view. Providers with a single feed type can auto-expand or skip straight to the form.

Trade-off: When multiple providers are expanded, the list gets long. Single-type providers (4 of 5 currently) make the accordion feel like unnecessary indirection — you expand just to see one option. Works better when providers have 2+ feed types each.

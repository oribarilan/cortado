Variant A: Hover-revealed arrows Recommended

  • + Zero new dependencies — pure state manipulation
  • + Minimal visual change from baseline — arrows only appear on hover
  • + Simple implementation — swap two array elements, call save
  • + Keyboard accessible via tab + enter on arrow buttons
  • + Clear affordance — arrows indicate direction unambiguously
  • - Slow for large reorders (move 1 position per click)
  • - Hidden on touch devices (hover-dependent) — but this is a desktop Tauri app

Best fit for Cortado: minimal surface area, no new deps, consistent with the app's "precision over decoration" principle. Most users have 3-8 feeds — single-position moves are fast enough.

Variant B: Drag handles

  • + Most intuitive reorder pattern — users expect drag-and-drop in list settings
  • + Fast for any reorder distance — move to any position in one gesture
  • + Grip icon (⠁⠁⠁) is a universally recognized drag affordance
  • - Requires a new dependency (@dnd-kit or similar) — project rules require user approval
  • - Complex implementation: drag state, drop zones, animated reflow, a11y announcements
  • - Conflicts with card click-to-edit — need to distinguish drag vs click on the handle area
  • - Accessibility: screen reader support for DnD is poor without significant ARIA work
  • - Handle coexists with the accent indicator bar — adds visual density to the left edge

Polished but heavy for the use case. Makes sense in apps with 20+ items or frequent reordering. For Cortado's typical 3-8 feeds configured once, it's over-engineered.

Variant C: Reorder mode toggle

  • + Clean separation — edit and reorder are distinct modes, no accidental reorders
  • + Position numbers give clear visual feedback of current order
  • + No new dependencies
  • + Arrow buttons are always visible in reorder mode — no hover discovery needed
  • - Extra mode to manage — adds state complexity (reorderMode boolean)
  • - More friction: click "Reorder" first, then move, then click "Done" — 3 steps vs 1
  • - Cards look different in each mode — the layout shift may feel jarring
  • - "Reorder" button in toolbar adds chrome; iOS-style pattern that feels out of place in a developer tool

Good for apps where accidental reordering would be disruptive. In Cortado, feed edits already require clicking into a card and saving — accidental reorder risk from hover-arrows is low.

Variant D: Keyboard-first with persistent handles

  • + Best accessibility — full keyboard control, focus ring, ARIA integration
  • + Handles always visible — no hover discovery problem
  • + Developer-friendly — Cmd+Arrow is a natural keyboard pattern
  • + No new dependencies
  • - Always-visible handles add permanent visual noise to every card
  • - Keyboard hint bar adds chrome above the list — justified only if keyboard is the primary input
  • - Cmd+Arrow may conflict with Tauri/OS shortcuts — needs testing
  • - Handle + accent indicator on left edge is dense; grip pattern competes with the feed indicator

Strong choice for power users. The keyboard hint bar adds discoverability but could be a tooltip instead. Consider combining with Variant A: hover-reveal arrows + keyboard shortcut support for moving focused cards.

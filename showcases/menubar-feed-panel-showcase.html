Fidelity Notes
  • Vibrancy: Native NSPanel uses macOS vibrancy/translucency material — cannot be replicated in HTML. Solid --surface used instead.
  • Panel arrow: Real panel has a system-drawn arrow attached to the tray icon; CSS triangle is approximate.
  • Scroll inertia: Native NSScrollView rubber-banding is not matched by browser overflow scroll.
  • Font weight 450: SF Pro supports optical weight 450; in non-macOS browsers this may round to 400.
  • oklch colors: Require modern browser (Chrome 111+, Safari 15.4+, Firefox 113+). Older browsers show fallback black/white.
  • Theme: "System" follows prefers-color-scheme; native app inherits directly from macOS appearance, which is equivalent.

Variant Notes & Tradeoffs

A — Minimal List

Extremely condensed single-line rows with a colored dot as the only status signal. Title takes maximum space; secondary fields are reduced to a single trailing number. Feed headers are de-emphasized uppercase labels.

+ Highest density. Scannable at a glance. Feels closest to native menubar extras (like Bartender, iStatMenus).
- Loses field detail: only one status + one number visible per row. Users must hover or click for context.

B — Signal Chips Emphasis

Two-line activity cards: title on top, then a full row of status/field chips below. Every field is rendered as a discrete colored chip, making signals like "review: awaiting" or "checks: failing" immediately visible with color coding.

+ Most informative — all fields visible without interaction. Status severity colors are prominent.
- Taller per activity — fewer activities visible in the panel. Can feel busy with many fields.

C — Grouped by Feed (Current, Refined)

Closest to the existing Cortado implementation. Single-row activities with inline field chips trailing after the title. Feed headers have name, type badge, and count. Balanced density and information.

+ Familiar layout. Good balance of info density vs. readability. Chip layout matches production code.
- Inline chips can get truncated on narrow panels. Middle-ground — neither most compact nor most detailed.

D — Dense Table Rows

Grid/table layout with fixed columns: Title, Review, Checks, and a metadata column. Column headers orient the user; every row aligns data vertically for fast comparison across activities.

+ Best for comparison across activities. Columnar alignment makes scanning status patterns easy.
- Rigid structure: different feeds may have different columns, requiring per-feed column definitions. Title column gets the least space.

E — Native Menu + Submenu Details

Activity rows look like native NSMenu items with a compact traffic-light status marker. Selecting an activity expands a submenu-style detail group that shows all fields. This keeps the top level very light and pushes detail to a second level.

+ Closest to native menubar menu mental model. Very scannable top level.
- Requires one extra interaction per activity to inspect details.

F — Native Menu + Inline Full Info

Every activity is rendered as a native-style menu row with inline full context (status + key fields). No submenu navigation; all data is visible directly in the list.

+ Zero-click details. Fast for power users who scan dense rows.
- Most visually dense native-menu option; can get noisy with long fields.

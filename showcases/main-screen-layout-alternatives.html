Main Screen — Layout Alternatives

The current split layout gives too little space to activity titles (which can be long PR names) while the detail pane often feels wasteful. These variants explore different approaches to balance title readability with detail access.

Problem: Activity titles (e.g., PR names like "feat(auth): implement OAuth2 PKCE flow with token refresh and session management") get heavily truncated in the 280px list pane. The detail pane takes ~50% of the window width but often shows only a few fields. The information density is low relative to the space used.

Current Implementation

Fixed 280px list + detail pane

The current layout. List pane is fixed at 280px — titles truncate early, especially for PR names. Detail pane fills the remaining ~280px.

  • Familiar split-pane pattern
  • Detail always visible without interaction
  • Titles heavily truncated — barely 30 characters visible
  • Detail pane often shows only 3-5 fields, wasting vertical and horizontal space
  • Fixed width doesn't adapt to content

Variant B

Wider list (60/40) with inline status chip

List takes 60% of width (~340px). Status chip moves inline to the list row, reducing the need to open the detail pane for quick triage. Detail pane becomes a secondary reference.

  • ~50% more characters visible in title
  • Status chip inline means you can triage without clicking each item
  • Detail pane still available for deep info
  • Narrower detail pane — field values may truncate on very long content
  • Chip takes some horizontal space from title

Variant C

Two-line rows with status + feed on second line

Each list row gets two lines: title on top, status chip + feed name on bottom. Gives the title the full row width while surfacing status without needing the detail pane.

  • Title gets the full 300px width — minimal truncation
  • Status visible at a glance without selecting
  • Feels richer and more informative per row
  • Fewer rows visible (roughly half as many in view)
  • May feel less compact for power users who scan by dot color alone

Variant D

Single pane with expandable detail (like the menubar panel)

No split. Activity titles get the full ~540px width. Click or press Enter to expand inline detail below the row, similar to how the menubar panel works. Maximizes title readability.

  • Titles get the full window width — essentially zero truncation
  • Consistent mental model with the menubar panel
  • Maximum information density when collapsed
  • Detail requires a click/keypress to reveal (not always-visible)
  • Expanding pushes other items down — context can shift
  • Less "app-like", more "panel-like"

Variant E

Flex split (60/40) with inline chip

Like Variant B but uses flex ratios instead of fixed widths, so the list grows with the window. Combines inline status chip with a proportional detail pane. A middle ground that adapts to different window sizes.

  • Proportional — list grows if user resizes window wider
  • Inline chip gives at-a-glance status
  • Detail pane still present for deep info
  • At minimum window size, both panes are somewhat cramped
  • Neither pane is truly "spacious"

Recommendation: Variant C (two-line rows) or Variant D (single pane expandable) best solve the core problem. Variant C preserves the split layout but gives titles the full list width. Variant D eliminates the wasted detail pane entirely and mirrors the menubar panel's proven expand/collapse pattern. Variant B is the least disruptive change from current — just widening the ratio and adding inline chips.
